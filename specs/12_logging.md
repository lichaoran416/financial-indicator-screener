## Topic 12: 后端日志规格 (Backend Logging)

### JTB-101: 记录API请求日志
**当** 用户发起API请求 **时**  
**我想要** 系统自动记录请求的详细信息（请求方法、路径、参数、耗时）  
**以便** 追踪用户行为和排查问题

### JTB-102: 记录错误日志
**当** API请求发生异常或错误 **时**  
**我想要** 系统记录完整的错误信息（错误类型、堆栈跟踪、请求上下文）  
**以便** 快速定位和修复问题

### JTB-103: 记录数据获取日志
**当** 系统从数据源（akshare）获取财务数据 **时**  
**我想要** 记录数据获取的请求详情和响应状态  
**以便** 监控数据源可用性和调试数据问题

### JTB-104: 日志分级管理
**当** 不同级别的日志事件发生时 **时**  
**我想要** 系统根据级别（DEBUG、INFO、WARNING、ERROR、CRITICAL）分别处理  
**以便** 在不同环境灵活控制日志详细程度

### JTB-105: 结构化日志输出
**当** 日志被记录 **时**  
**我想要** 日志以结构化格式（JSON）输出，包含时间戳、级别、模块、消息等标准字段  
**以便** 便于日志聚合和分析

### JTB-106: 请求追踪
**当** 处理一个API请求涉及多个服务调用 **时**  
**我想要** 通过唯一请求ID串联所有相关日志  
**以便** 完整追溯请求处理流程

### JTB-107: 敏感信息保护
**当** 记录日志时 **时**  
**我想要** 自动过滤或脱敏敏感信息（密码、Token、内部IP等）  
**以便** 防止敏感信息泄露

### JTB-108: 慢请求告警
**当** API请求处理时间超过预设阈值 **时**  
**我想要** 系统记录警告日志并触发告警  
**以便** 及时发现性能问题

### 日志配置要求
- **日志格式**: JSON结构化日志
- **日志级别**: DEBUG / INFO / WARNING / ERROR / CRITICAL
- **输出目标**: 标准输出（stdout）+ 文件存储
- **日志文件位置**: `logs/` （项目根目录下）
- **日志文件轮转**: 按日期切割，保留最近7天日志
- **请求ID**: UUIDv4格式，贯穿整个请求生命周期
- **性能指标**: 记录每个请求的处理时长（毫秒级精度）
